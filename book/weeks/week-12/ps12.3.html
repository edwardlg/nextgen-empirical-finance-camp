Part 1 — Dublin Core metadata (20 points)

The single sin this part prevents: a paper that ingests into MARS with incorrect metadata, which means search engines find it under the wrong query, which means the citing scholar of 2030 never finds it.

Dublin Core is the canonical metadata schema for institutional repositories. It has fifteen elements (Title, Creator, Subject, Description, Publisher, Contributor, Date, Type, Format, Identifier, Source, Language, Relation, Coverage, Rights). MARS uses the qualified Dublin Core variant, which allows refinements on elements (e.g., dc.date.issued, dc.date.accessioned, dc.date.available). Each element maps from a question your archive’s discovery interface will ask.

(a) Fill the fifteen Dublin Core elements (15 pts, ~1 each). For each element, write the value in metadata.md. Examples for Priya’s paper:

  • Title. “Flood-Map Revisions and Homeowner Insurance Take-Up: Evidence from a 2018 FEMA Boundary Update across 1,247 Counties.”
  • Creator. Each author as a separate dc.creator entry, format LastName, FirstName, MiddleInitial (e.g., “Singh, Priya N.”), with ORCID embedded.
  • Subject. Library-of-Congress Subject Headings (LCSH) — not JEL codes. For Priya: “Flood insurance — United States — Econometric models,” “Climate change — Economic aspects,” “Federal Emergency Management Agency.”
  • Description. The abstract, copied verbatim from PS 11.1.
  • Publisher. “George Mason University. Mason Archival Repository System.”
  • Contributor. The mentor (Prof. Lei Gao), with role “Thesis Advisor” or “Research Mentor.”
  • Date — issued. The date of deposit (e.g., 2026-09-25).
  • Date — accessioned. Filled by MARS at ingest; leave as [MARS-assigned].
  • Date — available. The date the public release begins (typically same as issued).
  • Type. “Working Paper” or “Preprint” — the canonical Dublin Core type for a non-journal-published research paper.
  • Format. “application/pdf” for the manuscript; “application/zip” for the replication package.
  • Identifier. MARS handle (assigned at ingest; leave as [MARS-assigned: hdl.handle.net/1920/xxxxx]).
  • Source. The SSRN URL (from PS 12.2) — establishes the parallel posting.
  • Language. “en” (ISO 639-1).
  • Relation. The journal submission (e.g., “isPartOf: Schar Young Scholars Journal, manuscript SYSJ-2026-0089”).
  • Coverage. Spatial and temporal extent of the empirical setting: “United States, 2014–2022.”
  • Rights. “CC BY 4.0” — Creative Commons Attribution 4.0 International (see Part 5).

The discipline: fill every element. An empty Dublin Core field is a discoverability hole.

(b) Generate the XML form (3 pts). Use the qualified Dublin Core XML schema:

<?xml version="1.0" encoding="UTF-8"?>
<metadata xmlns:dc="http://purl.org/dc/elements/1.1/" xmlns:dcterms="http://purl.org/dc/terms/">
  <dc:title>Flood-Map Revisions and Homeowner Insurance Take-Up: ...</dc:title>
  <dc:creator>Singh, Priya N.</dc:creator>
  <dc:creator>Gao, Lei</dc:creator>
  <dc:subject>Flood insurance — United States — Econometric models</dc:subject>
  ...
</metadata>

Validate the XML with xmllint --noout metadata.xml. The validator should report no errors.

(c) Map Dublin Core to MARC21 (2 pts). MARS internally stores records in MARC21 (the Library of Congress’s MAchine-Readable Cataloging format). Most Dublin Core elements map to standard MARC21 fields (e.g., dc.title → MARC21 245, dc.creator → 100/700, dc.subject → 650, dc.date.issued → 260$c). Produce a short mapping table in metadata.md showing your Dublin Core elements and their MARC21 targets. The library staff will do the actual mapping at ingest; documenting your intent prevents misinterpretation.

Part 2 — License and rights decisions (15 points)

The single sin this part prevents: choosing the wrong license, which either locks down your work (so future researchers cannot legally re-use it) or leaves it with no license (the default copyright restrictions, which are the most restrictive possible).

Code and data on MARS need an explicit open license. The default (no license) is all rights reserved, which means future researchers cannot legally re-distribute your code or build on it. The discipline is to choose a license and document the choice.

(a) Choose a license for the manuscript text (3 pts). Recommended: Creative Commons Attribution 4.0 International (CC BY 4.0). This license permits sharing and adaptation with attribution. Document in license.txt with the full license URL and a one-paragraph rationale. Why CC BY 4.0: (i) it is the SSRN-recommended preprint license, (ii) it is compatible with most journal copyright transfer agreements (you license the preprint; the journal gets the typeset version under their CTA), (iii) it permits re-use in derivative work with attribution, which is the goal.

(b) Choose a license for the code (4 pts). Recommended: MIT License or BSD 3-Clause. Both are permissive open-source licenses; both allow commercial re-use, modification, and re-distribution. Avoid GPL (copyleft) for empirical-research code — the copyleft requirement that derivative works also be GPL-licensed is incompatible with typical commercial replication contexts. Document the choice in replication-package/LICENSE.

(c) Choose a license / use terms for the derived data (4 pts). Recommended for aggregated, county-level, non-personally-identifiable derived data: CC0 (public domain dedication) — derived analytical data sets that contain no PII can be released into the public domain. Recommended for individual-loan-level or person-level derived data: do NOT redistribute. The HMDA-LAR raw data is already public, but any enriched version that links HMDA records to other identifiers becomes potentially identifying and should not be redistributed. State the choice explicitly in replication-package/DATA-LICENSE.md.

For Maya, whose paper uses HMDA-LAR (a public-use file), the derived county-level panel can be released under CC0. For Devon, whose paper uses public-blockchain on-chain data, the derived data is also CC0. For Priya, whose paper uses FEMA flood-map data and public NFIP claims data, the derived data is CC0. For Sam, whose paper uses minute-level CRSP intraday data (licensed via WRDS), the derived data — minute-level returns aggregated to event-time windows — is subject to CRSP’s redistribution restrictions; the deposit includes only the code, not the derived data, and the README documents how a future replicator with WRDS access can re-build the derived data. For Leah, whose paper uses USPTO patent data (public) plus 10-K filings (public), the derived data is CC0.

(d) Set the embargo (4 pts). MARS allows authors to embargo the public release of a deposit for up to 24 months (typical use: protect priority while a journal version is in review). For camp papers, the recommended choice is no embargo — the goal is the public record, not the protected priority. Document the no-embargo decision in metadata.md. If you do choose an embargo (e.g., because the journal review process specifically requires it), document the rationale and the end date.

Part 3 — The replication package (25 points)

The single sin this part prevents: a replication package that does not actually replicate. A future researcher tries to re-run your code, hits a missing dependency, a stale path, or an undocumented manual step, and gives up.

The replication package is the most consequential artifact of PS 12.3. Everything else (metadata, license, video) supports the package; the package is what enables reproducibility.

(a) Write the canonical README (8 pts). A single README.md at the root of the replication package, structured as follows:

# Replication Package: [Title]

## Overview
[1-paragraph description of the paper and what this package replicates.]

## Authors
- [Name 1], [affiliation], [email], [ORCID].
- [Name 2], [affiliation], [email], [ORCID].

## Citation
[Recommended citation sentence from PS 12.2 Part 5(c).]

## Software Requirements
- Python 3.11 (tested with 3.11.7).
- Operating system: Linux, macOS, or Windows with WSL2.
- Approximately [X] GB of disk space.
- Approximately [Y] minutes of compute time on a 2024-era laptop.

## Data Sources
- [Source 1, by proper noun]: [licensing], [where to obtain].
- [Source 2]: ...

## How to Reproduce
```bash
git clone [repo URL]
cd [repo]
conda env create -f environment.yml
conda activate [env-name]
bash reproduce.sh

This produces every table and figure in the paper, in out/, in approximately [Y] minutes.

File Manifest

[Tree of the package, with one-line description of each file.]

License

  • Code: MIT (see LICENSE).
  • Derived data: CC0 (see DATA-LICENSE.md).

Contact

[Corresponding author email.]


The README is the entry point. A reader who reads only the README and follows the "How to Reproduce" instructions should be able to reproduce the entire paper.

**(b) Write the master `reproduce.sh` script (10 pts).** A single shell script at the root of the package that, when invoked as `bash reproduce.sh`, runs the entire pipeline from raw data to final tables and figures. Required structure:

```bash
#!/usr/bin/env bash
set -euo pipefail
set -x  # Echo every command, for debugging.

# Step 0: environment check.
python -c "import sys; assert sys.version_info[:2] == (3, 11), 'Need Python 3.11'"

# Step 1: fetch raw data.
mkdir -p data/raw
python scripts/fetch_hmda.py  # downloads HMDA-LAR for 2014-2020
python scripts/fetch_census.py  # downloads county-level demographics
# ... (more fetch scripts as needed)

# Step 2: build analytical dataset.
python scripts/build_panel.py  # produces data/derived/county_panel.parquet

# Step 3: estimate models.
python scripts/estimate_did.py  # produces results/did_main.csv
python scripts/estimate_heterogeneity.py  # produces results/heterogeneity.csv

# Step 4: build tables.
python scripts/make_table1.py  # produces out/table1.tex
python scripts/make_table2.py  # produces out/table2.tex
python scripts/make_table3.py  # produces out/table3.tex

# Step 5: build figures.
python scripts/make_figure1.py  # produces out/figure1.pdf
python scripts/make_figure2.py  # produces out/figure2.pdf

# Step 6: build the response-to-tables-and-figures Markdown report.
python scripts/make_report.py  # produces out/report.md

echo "Reproduction complete. See out/ for tables and figures."

The script must run end-to-end without manual intervention. Test the script on a clean environment (a fresh conda env, or a separate machine, or a Docker container). If you cannot test on a separate machine, document the limitation and add a note: “tested on author’s own machine; clean-machine test pending.”

(c) Write a build manifest (4 pts). A separate file, replication-package/BUILD-MANIFEST.md, that lists every output artifact (table, figure, intermediate file) with its source script, its size, and its SHA-256 hash. The build manifest is the contract that says “running reproduce.sh produces exactly these files with exactly these hashes.” A future replicator can verify the output against the manifest and immediately know if anything has drifted.

(d) Include a Docker fallback (3 pts). A Dockerfile at the root of the package that, when built and run, reproduces the entire analysis. The Dockerfile freezes the OS, the Python version, and every dependency. Provide a one-paragraph note in the README on the Docker fallback for replicators who cannot get the conda environment working. Docker is the belt-and-suspenders version of reproducibility; the conda env is the typical path, Docker is the failsafe.

Part 4 — The reproducibility walkthrough video (15 points)

The single sin this part prevents: a replication package that is technically correct but pedagogically opaque — a stranger can run the code but cannot understand what it does.

The walkthrough video is the human face of the replication package. It is recorded once, hosted on MARS as a video file alongside the zip, and watched by every future replicator before they touch the code.

(a) Record an 8–12 minute screen-captured walkthrough (10 pts). Use OBS Studio, QuickTime, or a similar screen recorder. Cover, in order:

  • Minutes 0:00–1:30 — what the paper does. One-sentence description, the headline result with magnitude, the data, the design. The video opens with the paper, not the code.
  • Minutes 1:30–3:30 — the replication-package overview. Open the package in a file browser. Walk through the top-level structure: README, reproduce.sh, scripts/, data/, results/, out/, LICENSE. Name the role of each.
  • Minutes 3:30–6:00 — running reproduce.sh. Open a terminal, type bash reproduce.sh, and let it run. While it runs, narrate what each step is doing. Cut to the side any long fetch operations (or pre-record the fetched data and skip Step 1 for the recording).
  • Minutes 6:00–8:30 — inspecting the output. Open out/table2.tex in a viewer; show that the headline number matches the paper. Open out/figure1.pdf; show that it matches Figure 1 in the manuscript. The visual match is the credibility moment.
  • Minutes 8:30–10:00 — common modifications. Show how a future replicator could change one parameter (e.g., the bandwidth in a kernel, the clustering level, the sample years) and re-run. This is the “extension” demonstration.
  • Minutes 10:00–close — closing. Direct viewers to the MARS handle, the SSRN URL, and the corresponding author’s email for further questions.

Record at 1080p, mono audio (a USB headset works fine), no background music. The video does not need to be polished; it needs to be clear. If you record and re-watch and cringe, re-record once and move on; perfection here is the enemy of done.

(b) Write a timestamped transcript (5 pts). A Markdown file, walkthrough-transcript.md, that transcribes the video with timestamps every ~30 seconds. The transcript serves three audiences: (i) deaf and hard-of-hearing replicators who cannot hear the audio, (ii) replicators who want to search the video for a specific topic (“when does she talk about the data fetch?”), (iii) the future-you who wants to remember what was said without re-watching. The transcript is the search index.

Part 5 — Validation and discoverability audit (15 points)

The single sin this part prevents: shipping a deposit that ingests with errors, has malformed metadata, or fails to appear in Google Scholar search.

(a) Run the MARS metadata validator (5 pts). MARS publishes its metadata schema as an XSD (XML Schema Definition). Validate your metadata.xml against the schema:

xmllint --schema mars-schema.xsd metadata.xml --noout

(For the camp, use the published MARS XSD; in practice the file is downloaded from the MARS administrator’s documentation page.) Document the validator output — “validates” or a list of errors — in validator-output.md. If the validator reports errors, fix the metadata and re-run; the deposit is not ready until the validator reports clean.

(b) Run the OAI-PMH discoverability check (5 pts). Once MARS ingests, the metadata is published via OAI-PMH (Open Archives Initiative Protocol for Metadata Harvesting). Verify the record is discoverable by harvesters such as Google Scholar, BASE (Bielefeld Academic Search Engine), and OpenAIRE. For the camp, simulate the check: confirm that your metadata.xml would respond correctly to the canonical OAI-PMH verbs (Identify, ListMetadataFormats, GetRecord, ListRecords). The simulation lives in discoverability-check.md as a short table:

OAI-PMH verb Expected response Notes
Identify Returns MARS repository identity Confirmed by MARS docs
ListMetadataFormats Returns oai_dc (Dublin Core) Default for MARS
GetRecord?identifier=<handle>&metadataPrefix=oai_dc Returns this record Confirmed by sample query

(c) Run the Google Scholar indexing check (5 pts). Within 30 days of deposit, Google Scholar should index the MARS record. Document the indexing check in discoverability-check.md: the expected search query (your title, in quotes), the expected result (a Google Scholar entry pointing to your MARS handle), and the contingency plan if it does not index (manual submission to Google Scholar’s Webmasters interface — see Ch 12.1 §3 for the workflow). For the camp, the indexing happens after deposit, so the check is a forward-looking plan: “by 2026-10-25, the paper will appear in Google Scholar; if not, the contingency plan is …”

Part 6 — The deposit narrative and submission memo (10 points)

The single sin this part prevents: depositing the package without a human-readable accompanying note for the MARS librarian, who at ingest time may have questions that the metadata.xml does not answer.

(a) Write the deposit submission memo (7 pts). A one-page memo, mars-packet/deposit-memo.md, addressed to the MARS administrator (mars@gmu.edu). Required sections:

  • Subject: “MARS Deposit Request — [Title]”
  • Author and identifying information.
  • Description of the deposit. Manuscript + replication package + walkthrough video, ~[X] MB total.
  • Special instructions for ingestion. E.g., “video file should be transcoded to H.264 if your platform does not support the source codec,” “replication-package zip should be hosted at the same handle as the manuscript, not as a separate item.”
  • Requested embargo: None.
  • Requested license: CC BY 4.0 for manuscript; MIT for code; CC0 for derived data.
  • Corresponding author for archival questions.
  • Closing.

(b) Document the deposit timeline (3 pts). A short timeline in the memo naming when each step happens: deposit submitted (e.g., 2026-09-25), MARS administrator review (5 business days), ingestion (within 10 business days of approval), handle assignment (at ingestion), Google Scholar indexing (within 30 days), OAI-PMH harvest propagation (within 60 days). The timeline is the forward-looking calendar.

10. `mars-packet/discoverability-check.md` — the OAI-PMH and Google Scholar discoverability audit.

## Part 1 — Dublin Core metadata (20 points)

The single sin this part prevents: **a paper that ingests into MARS with incorrect metadata, which means search engines find it under the wrong query, which means the citing scholar of 2030 never finds it.**

Dublin Core is the canonical metadata schema for institutional repositories. It has fifteen elements (Title, Creator, Subject, Description, Publisher, Contributor, Date, Type, Format, Identifier, Source, Language, Relation, Coverage, Rights). MARS uses the qualified Dublin Core variant, which allows refinements on elements (e.g., `dc.date.issued`, `dc.date.accessioned`, `dc.date.available`). Each element maps from a question your archive's discovery interface will ask.

**(a) Fill the fifteen Dublin Core elements (15 pts, ~1 each).** For each element, write the value in `metadata.md`. Examples for Priya's paper:

- *Title.* "Flood-Map Revisions and Homeowner Insurance Take-Up: Evidence from a 2018 FEMA Boundary Update across 1,247 Counties."
- *Creator.* Each author as a separate `dc.creator` entry, format `LastName, FirstName, MiddleInitial` (e.g., "Singh, Priya N."), with ORCID embedded.
- *Subject.* Library-of-Congress Subject Headings (LCSH) — not JEL codes. For Priya: "Flood insurance — United States — Econometric models," "Climate change — Economic aspects," "Federal Emergency Management Agency."
- *Description.* The abstract, copied verbatim from PS 11.1.
- *Publisher.* "George Mason University. Mason Archival Repository System."
- *Contributor.* The mentor (Prof. Lei Gao), with role "Thesis Advisor" or "Research Mentor."
- *Date — issued.* The date of deposit (e.g., 2026-09-25).
- *Date — accessioned.* Filled by MARS at ingest; leave as `[MARS-assigned]`.
- *Date — available.* The date the public release begins (typically same as issued).
- *Type.* "Working Paper" or "Preprint" — the canonical Dublin Core type for a non-journal-published research paper.
- *Format.* "application/pdf" for the manuscript; "application/zip" for the replication package.
- *Identifier.* MARS handle (assigned at ingest; leave as `[MARS-assigned: hdl.handle.net/1920/xxxxx]`).
- *Source.* The SSRN URL (from PS 12.2) — establishes the parallel posting.
- *Language.* "en" (ISO 639-1).
- *Relation.* The journal submission (e.g., "isPartOf: Schar Young Scholars Journal, manuscript SYSJ-2026-0089").
- *Coverage.* Spatial and temporal extent of the empirical setting: "United States, 2014–2022."
- *Rights.* "CC BY 4.0" — Creative Commons Attribution 4.0 International (see Part 5).

The discipline: fill every element. An empty Dublin Core field is a discoverability hole.

**(b) Generate the XML form (3 pts).** Use the qualified Dublin Core XML schema:

```xml
<?xml version="1.0" encoding="UTF-8"?>
<metadata xmlns:dc="http://purl.org/dc/elements/1.1/" xmlns:dcterms="http://purl.org/dc/terms/">
  <dc:title>Flood-Map Revisions and Homeowner Insurance Take-Up: ...</dc:title>
  <dc:creator>Singh, Priya N.</dc:creator>
  <dc:creator>Gao, Lei</dc:creator>
  <dc:subject>Flood insurance — United States — Econometric models</dc:subject>
  ...
</metadata>
```

Validate the XML with `xmllint --noout metadata.xml`. The validator should report no errors.

**(c) Map Dublin Core to MARC21 (2 pts).** MARS internally stores records in MARC21 (the Library of Congress's MAchine-Readable Cataloging format). Most Dublin Core elements map to standard MARC21 fields (e.g., `dc.title` → MARC21 245, `dc.creator` → 100/700, `dc.subject` → 650, `dc.date.issued` → 260$c). Produce a short mapping table in `metadata.md` showing your Dublin Core elements and their MARC21 targets. The library staff will do the actual mapping at ingest; documenting your intent prevents misinterpretation.

## Part 2 — License and rights decisions (15 points)

The single sin this part prevents: **choosing the wrong license, which either locks down your work (so future researchers cannot legally re-use it) or leaves it with no license (the default copyright restrictions, which are the most restrictive possible).**

Code and data on MARS need an explicit open license. The default (no license) is *all rights reserved*, which means future researchers cannot legally re-distribute your code or build on it. The discipline is to *choose* a license and document the choice.

**(a) Choose a license for the manuscript text (3 pts).** Recommended: *Creative Commons Attribution 4.0 International* (CC BY 4.0). This license permits sharing and adaptation with attribution. Document in `license.txt` with the full license URL and a one-paragraph rationale. Why CC BY 4.0: (i) it is the SSRN-recommended preprint license, (ii) it is compatible with most journal copyright transfer agreements (you license the preprint; the journal gets the typeset version under their CTA), (iii) it permits re-use in derivative work with attribution, which is the goal.

**(b) Choose a license for the code (4 pts).** Recommended: *MIT License* or *BSD 3-Clause*. Both are permissive open-source licenses; both allow commercial re-use, modification, and re-distribution. Avoid GPL (copyleft) for empirical-research code — the copyleft requirement that derivative works also be GPL-licensed is incompatible with typical commercial replication contexts. Document the choice in `replication-package/LICENSE`.

**(c) Choose a license / use terms for the derived data (4 pts).** Recommended for *aggregated, county-level, non-personally-identifiable* derived data: *CC0* (public domain dedication) — derived analytical data sets that contain no PII can be released into the public domain. Recommended for *individual-loan-level or person-level* derived data: do NOT redistribute. The HMDA-LAR raw data is already public, but any *enriched* version that links HMDA records to other identifiers becomes potentially identifying and should not be redistributed. State the choice explicitly in `replication-package/DATA-LICENSE.md`.

For Maya, whose paper uses HMDA-LAR (a public-use file), the derived county-level panel can be released under CC0. For Devon, whose paper uses public-blockchain on-chain data, the derived data is also CC0. For Priya, whose paper uses FEMA flood-map data and public NFIP claims data, the derived data is CC0. For Sam, whose paper uses minute-level CRSP intraday data (licensed via WRDS), the *derived* data — minute-level returns aggregated to event-time windows — is subject to CRSP's redistribution restrictions; the deposit includes only the code, not the derived data, and the README documents how a future replicator with WRDS access can re-build the derived data. For Leah, whose paper uses USPTO patent data (public) plus 10-K filings (public), the derived data is CC0.

**(d) Set the embargo (4 pts).** MARS allows authors to embargo the public release of a deposit for up to 24 months (typical use: protect priority while a journal version is in review). For camp papers, the recommended choice is *no embargo* — the goal is the public record, not the protected priority. Document the no-embargo decision in `metadata.md`. If you do choose an embargo (e.g., because the journal review process specifically requires it), document the rationale and the end date.

## Part 3 — The replication package (25 points)

The single sin this part prevents: **a replication package that does not actually replicate. A future researcher tries to re-run your code, hits a missing dependency, a stale path, or an undocumented manual step, and gives up.**

The replication package is the most consequential artifact of PS 12.3. Everything else (metadata, license, video) supports the package; the package is what enables reproducibility.

**(a) Write the canonical README (8 pts).** A single `README.md` at the root of the replication package, structured as follows:

```markdown
# Replication Package: [Title]

## Overview
[1-paragraph description of the paper and what this package replicates.]

## Authors
- [Name 1], [affiliation], [email], [ORCID].
- [Name 2], [affiliation], [email], [ORCID].

## Citation
[Recommended citation sentence from PS 12.2 Part 5(c).]

## Software Requirements
- Python 3.11 (tested with 3.11.7).
- Operating system: Linux, macOS, or Windows with WSL2.
- Approximately [X] GB of disk space.
- Approximately [Y] minutes of compute time on a 2024-era laptop.

## Data Sources
- [Source 1, by proper noun]: [licensing], [where to obtain].
- [Source 2]: ...

## How to Reproduce
```bash
git clone [repo URL]
cd [repo]
conda env create -f environment.yml
conda activate [env-name]
bash reproduce.sh
```
This produces every table and figure in the paper, in `out/`, in approximately [Y] minutes.

## File Manifest
[Tree of the package, with one-line description of each file.]

## License
- Code: MIT (see LICENSE).
- Derived data: CC0 (see DATA-LICENSE.md).

## Contact
[Corresponding author email.]
```

The README is the entry point. A reader who reads only the README and follows the "How to Reproduce" instructions should be able to reproduce the entire paper.

**(b) Write the master `reproduce.sh` script (10 pts).** A single shell script at the root of the package that, when invoked as `bash reproduce.sh`, runs the entire pipeline from raw data to final tables and figures. Required structure:

```bash
#!/usr/bin/env bash
set -euo pipefail
set -x  # Echo every command, for debugging.

# Step 0: environment check.
python -c "import sys; assert sys.version_info[:2] == (3, 11), 'Need Python 3.11'"

# Step 1: fetch raw data.
mkdir -p data/raw
python scripts/fetch_hmda.py  # downloads HMDA-LAR for 2014-2020
python scripts/fetch_census.py  # downloads county-level demographics
# ... (more fetch scripts as needed)

# Step 2: build analytical dataset.
python scripts/build_panel.py  # produces data/derived/county_panel.parquet

# Step 3: estimate models.
python scripts/estimate_did.py  # produces results/did_main.csv
python scripts/estimate_heterogeneity.py  # produces results/heterogeneity.csv

# Step 4: build tables.
python scripts/make_table1.py  # produces out/table1.tex
python scripts/make_table2.py  # produces out/table2.tex
python scripts/make_table3.py  # produces out/table3.tex

# Step 5: build figures.
python scripts/make_figure1.py  # produces out/figure1.pdf
python scripts/make_figure2.py  # produces out/figure2.pdf

# Step 6: build the response-to-tables-and-figures Markdown report.
python scripts/make_report.py  # produces out/report.md

echo "Reproduction complete. See out/ for tables and figures."
```

The script must run end-to-end without manual intervention. Test the script on a *clean* environment (a fresh conda env, or a separate machine, or a Docker container). If you cannot test on a separate machine, document the limitation and add a note: "tested on author's own machine; clean-machine test pending."

**(c) Write a build manifest (4 pts).** A separate file, `replication-package/BUILD-MANIFEST.md`, that lists every output artifact (table, figure, intermediate file) with its source script, its size, and its SHA-256 hash. The build manifest is the contract that says "running `reproduce.sh` produces *exactly these files* with *exactly these hashes*." A future replicator can verify the output against the manifest and immediately know if anything has drifted.

**(d) Include a Docker fallback (3 pts).** A `Dockerfile` at the root of the package that, when built and run, reproduces the entire analysis. The Dockerfile freezes the OS, the Python version, and every dependency. Provide a one-paragraph note in the README on the Docker fallback for replicators who cannot get the conda environment working. Docker is the *belt-and-suspenders* version of reproducibility; the conda env is the typical path, Docker is the failsafe.

## Part 4 — The reproducibility walkthrough video (15 points)

The single sin this part prevents: **a replication package that is technically correct but pedagogically opaque — a stranger can run the code but cannot understand what it does.**

The walkthrough video is the human face of the replication package. It is recorded once, hosted on MARS as a video file alongside the zip, and watched by every future replicator before they touch the code.

**(a) Record an 8–12 minute screen-captured walkthrough (10 pts).** Use OBS Studio, QuickTime, or a similar screen recorder. Cover, in order:

- *Minutes 0:00–1:30 — what the paper does.* One-sentence description, the headline result with magnitude, the data, the design. The video opens with the paper, not the code.
- *Minutes 1:30–3:30 — the replication-package overview.* Open the package in a file browser. Walk through the top-level structure: README, reproduce.sh, scripts/, data/, results/, out/, LICENSE. Name the role of each.
- *Minutes 3:30–6:00 — running `reproduce.sh`.* Open a terminal, type `bash reproduce.sh`, and let it run. While it runs, narrate what each step is doing. Cut to the side any long fetch operations (or pre-record the fetched data and skip Step 1 for the recording).
- *Minutes 6:00–8:30 — inspecting the output.* Open `out/table2.tex` in a viewer; show that the headline number matches the paper. Open `out/figure1.pdf`; show that it matches Figure 1 in the manuscript. The visual match is the credibility moment.
- *Minutes 8:30–10:00 — common modifications.* Show how a future replicator could change one parameter (e.g., the bandwidth in a kernel, the clustering level, the sample years) and re-run. This is the "extension" demonstration.
- *Minutes 10:00–close — closing.* Direct viewers to the MARS handle, the SSRN URL, and the corresponding author's email for further questions.

Record at 1080p, mono audio (a USB headset works fine), no background music. The video does not need to be polished; it needs to be *clear*. If you record and re-watch and cringe, re-record once and move on; perfection here is the enemy of done.

**(b) Write a timestamped transcript (5 pts).** A Markdown file, `walkthrough-transcript.md`, that transcribes the video with timestamps every ~30 seconds. The transcript serves three audiences: (i) deaf and hard-of-hearing replicators who cannot hear the audio, (ii) replicators who want to search the video for a specific topic ("when does she talk about the data fetch?"), (iii) the future-you who wants to remember what was said without re-watching. The transcript is the search index.

## Part 5 — Validation and discoverability audit (15 points)

The single sin this part prevents: **shipping a deposit that ingests with errors, has malformed metadata, or fails to appear in Google Scholar search.**

**(a) Run the MARS metadata validator (5 pts).** MARS publishes its metadata schema as an XSD (XML Schema Definition). Validate your `metadata.xml` against the schema:

```bash
xmllint --schema mars-schema.xsd metadata.xml --noout
```

(For the camp, use the published MARS XSD; in practice the file is downloaded from the MARS administrator's documentation page.) Document the validator output — "validates" or a list of errors — in `validator-output.md`. If the validator reports errors, fix the metadata and re-run; the deposit is not ready until the validator reports clean.

**(b) Run the OAI-PMH discoverability check (5 pts).** Once MARS ingests, the metadata is published via OAI-PMH (Open Archives Initiative Protocol for Metadata Harvesting). Verify the record is discoverable by harvesters such as Google Scholar, BASE (Bielefeld Academic Search Engine), and OpenAIRE. For the camp, *simulate* the check: confirm that your `metadata.xml` would respond correctly to the canonical OAI-PMH verbs (`Identify`, `ListMetadataFormats`, `GetRecord`, `ListRecords`). The simulation lives in `discoverability-check.md` as a short table:

| OAI-PMH verb | Expected response | Notes |
|---|---|---|
| `Identify` | Returns MARS repository identity | Confirmed by MARS docs |
| `ListMetadataFormats` | Returns `oai_dc` (Dublin Core) | Default for MARS |
| `GetRecord?identifier=<handle>&metadataPrefix=oai_dc` | Returns this record | Confirmed by sample query |

**(c) Run the Google Scholar indexing check (5 pts).** Within 30 days of deposit, Google Scholar should index the MARS record. Document the indexing check in `discoverability-check.md`: the expected search query (your title, in quotes), the expected result (a Google Scholar entry pointing to your MARS handle), and the contingency plan if it does not index (manual submission to Google Scholar's `Webmasters` interface — see Ch 12.1 §3 for the workflow). For the camp, the indexing happens after deposit, so the check is a forward-looking plan: "by 2026-10-25, the paper will appear in Google Scholar; if not, the contingency plan is …"

The timeline is the forward-looking calendar.

## The bridge to PS 12.4 and beyond

PS 12.3 produces the *archival-permanent version of the paper*. PS 12.4 plans the eighteen-month conference circuit that will move the paper through Tier 5 → Tier 4 → Tier 3 venues. PS 12.5 zooms to the five-year identity arc. The MARS deposit is the single record that, in 2034, will still exist in its 2026 form — the others (journal, SSRN, GitHub) may not. Treat MARS as the durable-record commitment it is.

*End of PS 12.3. The reproducibility discipline is not "your code runs"; it is "a stranger five years from now can run your code without asking you a question." That is the bar. Begin with the README and walk outward; the rest of the package supports the README's promise. By the end of the week the deposit will be submitted, the handle will be requested, and the walkthrough video will be sitting on disk next to the zip — ready to outlive every other version of the paper.*
